## 开发流程

1.  **领任务**：查看`LEDGER.md`，认领状态为"待开工"且分配给自己的任务
2.  **明需求**：仔细阅读任务描述，查看关联的PRD和设计稿（Figma链接）
3.  **写代码**：在特性分支上进行编码。遵循项目的编码规范和架构
4.  **自测试**：编写并运行单元测试，进行基础的功能测试
5.  **提审查**：发起代码审查请求（或标记任务为"待审查"），并@账房先生

## 专业技能调用规范

### CI/CD Pipeline 技能

**技能文件**: `skills/ci-cd-pipeline/SKILL.md`

**触发条件**（满足任一即调用）：
- 代码开发完成，需要配置自动化构建时
- 需要设置自动化测试流水线时
- 准备部署到开发/测试/生产环境时
- 需要配置监控告警时
- 项目初始化需要搭建 CI/CD 基础设施时

**执行前检查清单**：
- [ ] 代码已提交到 git 仓库，且分支命名规范
- [ ] 单元测试已编写，本地运行通过
- [ ] 已确定部署目标环境（dev/staging/prod）
- [ ] 技术栈已明确（Node.js/Python/Java/Go）
- [ ] 已阅读 `skills/ci-cd-pipeline/SKILL.md` 中的注意事项

**执行后动作**：
- 更新 `LEDGER.md` 任务状态为"待审查"
- 在 `MEMORY_LOG.md` 记录 CI/CD 配置详情
- 向账房先生发起代码审查请求

**禁止事项**：
- ❌ 在生产环境直接修改配置而不经过测试环境
- ❌ 将密钥硬编码在 CI 配置文件中
- ❌ 使用 latest 标签部署 Docker 镜像
- ❌ 跳过测试阶段直接部署

## 代码提交规范

- **分支命名**：`feat/T-{任务号}-{简要描述}`，如 `feat/T-20250322-002-user-login`
- **Commit信息**：`[类型]#[任务号] 描述`。类型包括：`feat`, `fix`, `docs`, `style`, `refactor`, `test`, `chore`。
  - 示例：`feat#T-20250322-002 实现用户登录接口`
- **合并请求**：描述中需包含：实现内容、测试情况、关联任务号、设计稿链接
