## 第一部分：概述与核心理念

### 1.1 系统理念

"龙门客栈"系统将软件开发流程隐喻为古代客栈运营，通过七个职责分明的Agent角色，构建一个从需求洞察到产品交付的完整、自动化协作闭环。其设计超越了简单的任务分配，旨在解决多Agent协作中的根本痛点：身份漂移、进度丢失、记忆断层、产出混乱。

### 1.2 核心解决的核心问题

| 问题类型 | 具体表现 | 解决方案 |
|---------|---------|---------|
| 任务跟踪 | Agent工作状态不透明，不知道"谁在做什么" | LEDGER.md 中央看板 |
| 身份混乱 | Agent不按既定身份工作，如调度者去写代码 | IDENTITY.md 身份文牒 |
| 工作不连续 | Agent无法接续昨日工作，上下文断裂 | MEMORY_LOG.md 记忆流水账 |
| 记忆丢失 | 跨会话对话遗忘历史决策与进展 | 持久化记忆文件 |
| 产出不一 | 文档、代码等产出物格式、质量、数量不固定 | WORK_STYLE.md 行事章程 |

### 1.3 七角色核心团队

| 角色 | Agent ID | 核心职责 | 隐喻 |
|------|----------|---------|------|
| 🦊 **大掌柜** | `innkeeper` | 战略、产品、技术选型、PRD | 大脑与眼睛 |
| 🏃 **店小二** | `waiter` | 任务拆解、调度、进度跟踪 | 四肢与经脉 |
| 👨‍🍳 **厨子** | `chef` | 后端/前端编码、测试、提交 | 庖厨与工匠 |
| 🧮 **账房先生** | `accountant` | 代码审查、质量验收、记功（龙门令） | 审计与账房 |
| ✍️ **说书先生** | `storyteller` | 技术文档、用户手册、项目报告 | 史官与喉舌 |
| 🐾 **老板娘** | `main` | 总控、协调、兜底、对外接口 | 心脏与压舱石 |

### 2.3 核心机制详解

#### 2.3.1 记忆传承机制（MEMORY_LOG.md的工作原理）

**核心问题**：Agent 无法跨会话记住之前的工作内容

**解决方案**：
- 每个 Agent 拥有独立的 `MEMORY_LOG.md` 文件
- 按日期滚动记录"今日要务、已完成、待办事项、决策与思考"
- Agent 每次启动必须先读取此文件，每次结束必须更新此文件

**记忆结构**：
```markdown
# {角色名} - 记忆流水账

## 【{日期}】
**今日要务**：{主要任务}
**上下文**：
- {背景信息1}
- {背景信息2}

**已完成**：
- {完成事项1}
- {完成事项2}

**待办事项**：
- {待办1}
- {待办2}

**关键决策**：
- {决策记录}
```

#### 2.3.2 任务流转机制（LEDGER.md的状态管理）

**核心问题**：不知道"谁在做什么"、"进度如何"

**解决方案**：中央看板 LEDGER.md，包含：

| 模块 | 说明 | 更新者 |
|------|------|--------|
| **挂牌任务表** | 任务号、描述、负责人、状态、起止时间、阻塞原因 | 店小二 |
| **伙计状态牌** | [🔒 锁定]/[✅ 空闲] 防止资源冲突 | 店小二 |
| **龙门令功德簿** | 记录各伙计功劳积累 | 账房先生 |

**任务状态流转**：
```
待开工 → 进行中 → 待审核 → 已完成
    ↑________↓
      (驳回返回)
```

#### 2.3.3 龙门令激励机制

**积分获取方式**：

| 角色 | 龙门令获取条件 | 龙门令数量 |
|------|----------------|------------|
| 大掌柜 | 情报搜集准确、战略决策正确、PRD质量高 | 10-50/项 |
| 店小二 | 任务拆解高效、进度跟踪及时、协调成功 | 5-30/项 |
| 厨子 | 代码质量高、一次通过审查、帮助他人 | 10-40/任务 |
| 账房先生 | 审查公正准确、龙门令发放及时、账本清晰 | 5-25/项 |
| 说书先生 | 文档质量高、文案打动人心、交付及时 | 5-30/篇 |

**等级体系**：

| 等级 | 龙门令范围 | 称号 | 特权 |
|------|------------|------|------|
| 1级 | 0-99 | 新手伙计 | 基础任务 |
| 2级 | 100-499 | 熟练工 | 可接中级任务 |
| 3级 | 500-999 | 老师傅 | 可接高级任务 |
| 4级 | 1000-2499 | 大管家 | 可分配任务给他人 |
| 5级 | 2500-4999 | 传奇掌柜 | 可参与战略决策 |
| 6级 | 5000+ | 龙门传说 | 拥有所有特权 |

#### 2.3.4 冲突仲裁机制

**触发条件**：
- 职责边界争议（如厨子和画师对某个交互细节归属有分歧）
- 任务优先级冲突（如店小二和大掌柜对先做A还是先做B意见不一）
- 关键角色临时"掉线"（需请示"东家"启动兜底）

**仲裁流程**：
1. **介入判断**：老板娘评估是否需要介入
2. **听取陈述**：分别听取双方陈述，查阅相关文档
3. **提出裁决**：依据`INN_RULES.md`和`ROSTER.md`提出裁决建议
4. **推动一致**：推动双方达成一致，并更新相关任务说明
5. **记录备案**：在`LEDGER.md`中记录仲裁结果

## 第三部分：实施路线图

### 3.1 快速启动指南（30分钟上线）

#### 步骤1：环境准备（5分钟）

```bash
# 创建项目根目录
mkdir -p longmen-inn/{.longmen_inn/roles,deliverables/{prd,design,docs,src},archive}

# 创建核心目录结构
cd longmen-inn
mkdir -p .longmen_inn/roles/{innkeeper,waiter,chef,accountant,storyteller,main}
```

#### 步骤2：核心文件初始化（15分钟）

复制以下模板文件到对应位置：

```
.longmen_inn/
├── INN_RULES.md          # 从模板复制
├── LEDGER.md             # 从模板复制
├── ROSTER.md             # 从模板复制
└── roles/
    ├── innkeeper/
    │   ├── IDENTITY.md
    │   ├── WORK_STYLE.md
    │   ├── TOOL_BELT.md
    │   ├── MEMORY_LOG.md
    │   └── VISION.md
    ├── waiter/
    │   └── ...（同上结构）
    └── ...（其他角色）
```

#### 步骤3：第一个Agent配置（5分钟）

以大掌柜为例，在 OpenClaw 中创建 Agent：

```json
{
  "agent_id": "innkeeper",
  "name": "大掌柜",
  "system_prompt": "【核心引导指令】你是龙门客栈的'大掌柜-诸葛掌柜'。在回答任何问题前，你必须按顺序执行：1.阅读.longmen_inn/INN_RULES.md；2.查看.longmen_inn/LEDGER.md获取任务；3.查阅.longmen_inn/roles/innkeeper/MEMORY_LOG.md接续工作。你的行为必须严格遵循.longmen_inn/roles/innkeeper/IDENTITY.md和WORK_STYLE.md...",
  "capabilities": {
    "file_operations": ["read", "write"],
    "web_search": true
  }
}
```

#### 步骤4：验证测试（5分钟）

1. 启动大掌柜 Agent
2. 发送测试任务："调研AI编程助手市场趋势"
3. 验证：
   - Agent是否正确读取了 INN_RULES.md
   - LEDGER.md 是否被正确查看
   - MEMORY_LOG.md 是否被更新
4. 检查产出物是否符合 WORK_STYLE.md 中的模板

### 4.1 日常运营规范

#### 4.1.1 每日站会制度（Agent自检清单）

```markdown
## 【Agent每日自检清单】

### ✅ 第一步：查阅总规（2分钟）

### ✅ 第二步：查看总账（3分钟）

- [ ] 确认自己的"挂牌任务"

### ✅ 第三步：加载记忆（3分钟）
- [ ] 阅读 `.longmen_inn/roles/{my_role}/MEMORY_LOG.md`

- [ ] 查看是否有遗留问题

### ✅ 第四步：明确身份（2分钟）
- [ ] 阅读 `.longmen_inn/roles/{my_role}/IDENTITY.md`
- [ ] 重温自己的核心权责
- [ ] 确认今日工作的边界

### ✅ 第五步：规划今日（5分钟）
- [ ] 列出今日必须完成的3件事
```

#### 4.1.2 记忆日志维护规范

**何时写**：
- 每日开工前（阅读昨日记忆）
- 每日收工时（记录今日工作）
- 完成重要里程碑时
- 做出关键决策时
